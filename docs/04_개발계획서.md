## 1. 기술 아키텍처

### 1.1 시스템 개요

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Client     │     │   Server     │     │  External    │
│  (Next.js)   │────▶│  (Next.js    │────▶│  Services    │
│              │◀────│   API Route) │◀────│              │
└──────────────┘     └──────┬───────┘     └──────────────┘
                            │
                     ┌──────┴───────┐
                     │              │
              ┌──────┴──┐   ┌──────┴──┐
              │ Saju    │   │   DB    │
              │ Engine  │   │(Supabase│
              │         │   │/Postgres│
              └─────────┘   └─────────┘
```

### 1.2 기술 스택

| 계층 | 기술 | 선정 이유 |
|------|------|----------|
| **프론트엔드** | Next.js 14+ (App Router) | SSR/SEO 지원, 모바일 웹 최적화 |
| **스타일링** | Tailwind CSS + shadcn/ui | 빠른 UI 구현, 반응형 지원 |
| **백엔드** | Next.js API Routes | 프론트와 통합, 서버리스 배포 용이 |
| **데이터베이스** | Supabase (PostgreSQL) | 무료 티어, Auth 내장, 실시간 기능 |
| **AI** | Claude API (Anthropic) | 한국어 해석 품질, 가드레일 지원 |
| **사주 엔진** | 자체 구현 (TypeScript) | 만세력/오행 계산 커스텀 로직 필요 |
| **결제** | 토스페이먼츠 또는 포트원 | 한국 간편결제 지원, 빠른 연동 |
| **배포** | Vercel | Next.js 최적화, 자동 배포 |
| **분석** | Google Analytics 4 + Mixpanel | 퍼널 분석, 이벤트 트래킹 |
| **모니터링** | Sentry | 에러 트래킹, 성능 모니터링 |

### 1.3 아키텍처 원칙
- **사주 계산과 AI 문장 생성 분리**: 계산 엔진은 결정적(deterministic), AI는 생성적
- **프롬프트를 템플릿 레이어로 관리**: 빈번한 수정에 대응
- **서버리스 우선**: 초기 인프라 비용 최소화
- **점진적 복잡성**: MVP는 단순하게, 검증 후 확장

## 3. 데이터베이스 설계

### 3.1 ERD 개요

```
users
  ├── saju_profiles (1:N)
  ├── results (1:N)
  │     └── questions (1:N)
  ├── reports (1:N)
  └── payments (1:N)
```

### 3.2 테이블 정의

#### users
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK) | Supabase Auth UID |
| created_at | timestamptz | 가입일 |
| updated_at | timestamptz | 수정일 |

#### saju_profiles
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK) | |
| user_id | uuid (FK) | users.id (nullable, 비회원 허용) |
| session_id | text | 비회원 세션 식별자 |
| calendar_type | enum | 'solar' / 'lunar' |

## 5. 사주 계산 엔진 설계

### 5.1 핵심 모듈

#### calendar.ts — 음양력 변환
- 양력 → 음력, 음력 → 양력 변환
- 윤달 처리
- 1940~2010년 범위 지원
- 참조: 한국천문연구원 만세력 데이터

#### pillars.ts — 사주 팔자 계산
- 년주 계산: 입춘 기준 연도 판별
- 월주 계산: 절기 기준 월 판별
- 일주 계산: 일진표 기반
- 시주 계산: 일간 기준 시간 천간 결정

#### five-elements.ts — 오행 분석
- 천간/지지별 오행 매핑
- 오행 개수 및 비율 산출
- 일간 강약 판단
- 상생/상극 관계 분석

#### daily-fortune.ts — 오늘의 운세 계산
- 오늘 일진 산출
- 사주 원국과 오늘 일진의 관계 분석
- 3축(연애/일/재물) 흐름 판단

### 5.2 테스트 전략
- 알려진 사주 데이터 100건 이상으로 계산 정확도 검증
- 경계값 테스트 (자정, 입춘 전후, 윤달 등)
- 출생시간 "모름" 케이스 검증

## 6. AI 프롬프트 설계

### 6.1 프롬프트 구조

```
[시스템 프롬프트]
  - 역할 정의 (사주 해설사)
  - 톤앤매너 가이드라인
  - 금지 사항 (가드레일)
  - 출력 형식 지정

[사주 데이터]
  - 사주 팔자 구조
  - 오행 분석 결과
  - 오늘 일진과의 관계
  - 일간 특성

[사용자 컨텍스트]
  - 성별
  - (추가질문 시) 사용자 질문 내용

[출력 지시]
  - 한 줄 총평
  - 3축 해석
  - 행동 조언
  - 행운 단서
```

### 6.2 프롬프트 관리 원칙
- 프롬프트 버전 관리 (v1, v2, ...)
- 수정이 잦은 카피는 프롬프트보다 템플릿 레이어에서 관리
- A/B 테스트를 위한 프롬프트 분기 지원

## 8. 개발 환경 및 배포

### 8.1 개발 환경
```bash
# 필수 도구

Git

pnpm build        # 프로덕션 빌드
pnpm test         # 테스트 실행
pnpm lint         # 린트 검사
```

### 8.2 환경 변수
```
# .env.local

TOSS_CLIENT_KEY=
TOSS_SECRET_KEY=
NEXT_PUBLIC_GA_ID=
MIXPANEL_TOKEN=
SENTRY_DSN=
```

### 8.3 환경 구성

#### 3개 환경 (Local / QA / Live)

| 환경 | URL | 용도 | 트리거 |
|------|-----|------|--------|
| **Local** | `localhost:3000` | 개발 및 즉시 확인 | `pnpm dev` |
| **QA (Preview)** | `*.vercel.app` (PR별 자동 생성) | 기능 검증, 테스트 | PR 생성 시 자동 배포 |
| **Live (Production)** | `ningning.vercel.app` (추후 커스텀 도메인) | 실제 서비스 운영 | `main` 브랜치 push 시 자동 배포 |

#### 환경별 특징

**Local (로컬 개발)**
- `pnpm dev`로 실행, 코드 변경 시 즉시 반영 (Hot Reload)
- `.env.local`에서 환경변수 로드
- API 키, DB 연결 등 개발용 설정 사용
- 빌드 확인: `pnpm build` → 에러 없으면 PR 생성 가능

**QA (Preview — Vercel)**
- GitHub에 PR을 생성하면 Vercel이 자동으로 Preview 환경을 배포
- PR마다 고유 URL 생성 (예: `ningning-abc123.vercel.app`)
- Live와 동일한 인프라에서 실행되므로 실제와 같은 조건으로 테스트 가능
- PR에 Preview URL이 자동 코멘트로 달림
- 환경변수는 Vercel 프로젝트 설정에서 "Preview" 환경에 별도 지정 가능

**Live (Production — Vercel)**
- `main` 브랜치에 push/merge 시 자동 배포
- 현재 URL: Vercel 기본 도메인 (추후 커스텀 도메인 연결)
- 환경변수는 Vercel 프로젝트 설정에서 "Production" 환경에 지정
- 롤백: Vercel 대시보드에서 이전 배포로 즉시 롤백 가능

### 8.4 배포 파이프라인

```
[Local 개발]
    ↓ git push (feature 브랜치)
[PR 생성]
    ↓ 자동
[QA — Vercel Preview 배포] ← Preview URL로 테스트
    ↓ 검증 완료 후
[main 머지]
    ↓ 자동
[Live — Vercel Production 배포]
```

#### 브랜치 전략
| 브랜치 | 용도 | 배포 대상 |
|--------|------|----------|
| `main` | 프로덕션 코드 | Live (자동 배포) |
| `feature/*` | 기능 개발 | PR 생성 시 QA Preview |
| `fix/*` | 버그 수정 | PR 생성 시 QA Preview |
| `hotfix/*` | 긴급 수정 | PR → main 직접 머지 → Live |

#### 배포 체크리스트
- [ ] 로컬에서 `pnpm build` 성공 확인
- [ ] TypeScript 타입 에러 없음 (`pnpm exec tsc --noEmit`)
- [ ] PR 생성 → Preview URL에서 기능 동작 확인
- [ ] main 머지 → Live 배포 확인

### 8.5 환경변수 관리

| 변수 | Local | QA (Preview) | Live (Production) |
|------|-------|--------------|--------------------|
| `ANTHROPIC_API_KEY` | `.env.local` | Vercel 환경변수 (Preview) | Vercel 환경변수 (Production) |
| `NEXT_PUBLIC_SUPABASE_URL` | `.env.local` | Vercel 환경변수 | Vercel 환경변수 |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `.env.local` | Vercel 환경변수 | Vercel 환경변수 |

**주의사항:**
- `.env.local`은 git에 커밋하지 않음 (`.gitignore`에 포함)
- Vercel 환경변수는 대시보드 → Settings → Environment Variables에서 관리
- Preview와 Production에 다른 값을 설정할 수 있음 (예: 테스트용 DB vs 운영 DB)

### 8.6 CI/CD

**CI (자동 검증)** — 추후 GitHub Actions 설정
- 린트 검사 (ESLint)
- 타입 체크 (TypeScript)
- 단위 테스트 (Vitest)
- 사주 엔진 정확도 테스트

**CD (자동 배포)** — Vercel 연동 완료
- PR 생성 → Preview 자동 배포 ✅
- main push → Production 자동 배포 ✅
- 롤백: Vercel 대시보드에서 클릭 1번으로 이전 버전 복원

### 8.7 장애 대응

| 상황 | 대응 |
|------|------|
| Live 배포 후 버그 발견 | Vercel 대시보드에서 이전 배포로 즉시 롤백 |
| AI API 장애 | 사주 계산 결과만 표시 (AI 해석 없이 엔진 기본 결과 fallback) |
| AI API 비용 급증 | Vercel 환경변수에서 API 키 비활성화 → 엔진 결과만 제공 |
| 트래픽 급증 | Vercel 서버리스 자동 스케일링 (별도 조치 불필요) |
